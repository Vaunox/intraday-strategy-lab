# MASTER BLUEPRINT — Intraday Strategy Research Lab
### Build & Research Handoff Document

*A single, self-contained specification for a **new repository** whose purpose is to build an honest research apparatus and then systematically test a fixed slate of classic intraday trading strategies on Indian liquid cash equities. The only data source is **Zerodha Kite Connect historical candle data**. The output is a validated verdict (trade / don't-trade) per strategy and a written research paper of findings.*

*This document is written to be handed to a capable engineer or coding agent and worked through top-to-bottom, **one full phase per session.***

## HOW TO USE THIS DOCUMENT

This file is the operating manual for the build. Work it **top to bottom, one full phase per session** (not one small subtask per session — a whole phase, start to gate, is the session unit).

**Two ordering rules govern everything below:**
1. **Build the research apparatus before testing any strategy.** Phases 0–2 build the honest machine that judges a strategy (foundation → data & features → validation harness). No strategy is run until that machine exists and its gate has passed. You cannot get an honest verdict from a harness that does not yet enforce point-in-time correctness, full costs, and the kill-gate.
2. **One full phase per session.** Finish an entire phase — every task, its tests, its gate — in a single session before moving on.

**At the start of every session:**
> "Follow the Engineering Ground Rules (Part I). We are working on **Phase [N]**. Read its referenced technical section and deep dives, build every task in the phase to its acceptance criteria, write the tests, run the quality gates, update the Progress Log (Part VI), and — for any strategy study — update the Research Paper (`docs/RESEARCH_FINDINGS.md`) with the real result."

**Rules of engagement for the whole build:**
- **One full phase per session — with one carve-out for the study phases.** A phase is the session unit, start to gate. The infrastructure phases (0, 1, 2, 5) are each completed in a single session: build all tasks, write the tests, pass the gate. **The study phases (3 and 4) are the exception:** running 14 (or 6) *full* validated studies — each with pre-registration, spec, the complete CPCV/DSR/PBO/robustness/two-engine battery, and a written paper section — in a single sitting collides directly with "never rush a verdict," which always wins. So Phase 3/4 may be **batched by strategy category** (e.g., all breakout studies in one session, all mean-reversion in the next), completing whole categories at a time against the same unchanged harness. The gate for the phase closes only when every study in it has an honest verdict — but the work may span sessions **by category**, never by leaving a single study half-validated. Do not stop mid-study; do not rush a verdict to close a batch. Depth wins over speed, always.
- **Respect dependencies and phase gates.** Do not start a phase until the previous phase's gate passes. The gates exist for safety and scientific honesty, not bureaucracy.
- **The research paper is a living deliverable.** The engineer/agent building each strategy study writes and updates `docs/RESEARCH_FINDINGS.md` with that study's real numbers and verdict *as part of the study* — it is never left for later and never pre-filled with imagined results.
- **Update the Progress Log (Part VI)** at the end of each session.
- When a decision is ambiguous, choose the **simpler, more robust, more testable** option. This is the recurring principle of the entire design.
- **Attribution — this is the operator's own work.** Commits, PRs, docs, code, comments, and repository metadata carry **no third-party or AI tool attribution of any kind** — no "co-authored-by", no "generated by", no "assisted by", no assistant, model, or vendor names, anywhere in the repository, its git history, its tags, or its metadata. The repository must read, in full, as the operator's own research project. This is a hard rule, checked before every commit.

# PART I — ENGINEERING GROUND RULES

Non-negotiable standards for every line of code. Re-read before each phase.

## 1. Modularity
Every component has one clear responsibility and can be tested and replaced independently.
- **One module = one job.** Data ingestion, feature computation, strategy specification, validation, cost modeling, and reporting each live in their own package with a clear boundary.
- **Program to interfaces, not implementations.** Define `Protocol`/abstract interfaces for anything with more than one possible implementation — the broker (`BrokerAdapter`), the storage layer (`Repository`), the strategy (`StrategySpec`). Nothing outside `data/brokers/` may import the broker SDK directly; nothing outside `data/store/` may import the storage client directly.
- **Dependency injection.** Pass dependencies in; do not reach for globals or construct collaborators inside business logic.
- **No global mutable state.** State lives in explicit, owned objects.
- **Pure functions where possible**, especially feature computation and strategy signals — same inputs, same output, no side effects. This is what makes point-in-time correctness enforceable.

## 2. Avoid Hard-Coding
No magic numbers, paths, credentials, thresholds, or environment assumptions in code.
- **All parameters live in versioned configuration** (`config/`), loaded at runtime: the universe, decision frequency, indicator lookbacks, barrier multiples, cost-model rates, file paths, rate limits — everything.
- **Layered config:** `default.yaml` base, overridden per environment, overridden by environment variables. One source of truth; every run reproducible from its config.
- **Secrets never in code or config.** API keys/tokens come only from a secrets interface / environment variables. Never committed, never logged.
- **OS-portable code, always.** Use `pathlib.Path`; never hardcode path separators. Code must run identically on the CI Linux container and the operator's dev machine.

Solve the underlying problem, not the symptom. No band-aids, no commented-out code, no "TODO: fix later" in merged work. If a value is occasionally wrong, find why. Refactor when the design strains rather than piling on special cases. A correct, slightly slower solution beats a fast, fragile one.

Comment the *why*, not the *what*. Docstrings on every public module, class, and function (purpose, params, returns, invariants — especially "feature/signal functions are point-in-time: inputs are only data available at or before `asof`"). Cite the technique where logic implements one (e.g., "triple-barrier labeling; see docs/deep_dives/02"). Keep comments truthful — update them when code changes.

- **No tool, AI, or authorship attribution in commit messages, PR bodies, tags, comments, or code.** No "co-authored-by", no "generated by", no assistant/model/vendor names. History reads as the operator's own work, end to end.

## 7. Professional Standards
Clean, readable code; clear names; small functions; formatter/linter enforced (ruff + black). **Static typing everywhere** — `mypy`/`pyright` must pass in CI. Every PR reviewed against acceptance criteria and these rules. **Determinism:** seed all RNGs; version data/features/results; any run reproducible from config + versions. **Fail loudly and early:** validate inputs at boundaries; raise clear, specific exceptions.

## 8. Proper Logging
Structured logging (not bare prints), configured once in `core/`. Appropriate levels (DEBUG/INFO/WARNING/ERROR/CRITICAL). Log every study run, config, and result. Correlation IDs to trace a run. Never log secrets. Timestamps in IST.

## 9. Completion Standard — "done" means proven at the call site, not "the happy path passes"
Established by the Phase-2→3 audit arc (`docs/PROGRESS.md` → "Phase 3 readiness"). Its finding: every defect had the same shape — a **name, flag, or tag claiming more than the code delivered** (a CV that didn't purge, a gate that graded stubs, tags predating their deliverables, a "harmless" caveat that wasn't). None computed a wrong answer; all were **accounting gaps between label and behavior**. These rules close that gap and bind Phases 3+.

1. **Definition of Done — both must hold.**
   - **(a) Claimed at a call site.** Every capability a function/module *name or docstring* claims is **invoked at a real call site** (cite `file:line` of the call), not merely defined. A primitive that exists but nothing calls is **not done** (the orphaned-`PurgedKFold` / un-wired-purge failure mode).
   - **(b) Certified against real machinery.** The certifying test feeds **real machinery-computed inputs** — never stubs, sentinels, or hand-assembled scalars. **Required property: the certifying test must fail if the underlying machinery is removed.** A test that would still pass with the machinery deleted certifies nothing (the gate2-stub failure mode).
2. **Names and docstrings are claims; they must be true at HEAD.** If a name asserts a property (*purged, size-aware, isolated, adaptive, shared*), the code must have it on the call path — or the name/docstring is corrected to what the code actually does. No aspirational naming.
3. **Gate/version tags are deliverable snapshots.** The tag commit must contain the **complete** deliverables the gate claims. Tagging ahead of deliverables is prohibited (as happened with `gate-1-data` / `gate-2-harness`). If work lands post-tag, re-cut the tag, or mark it non-snapshot in `PROGRESS.md` with **HEAD named as the source of truth**.
4. **Fail-closed is the default for every evaluator.** A gate refuses to grade (`INSUFFICIENT` / raise) on absent, sentinel, or un-provenanced input — it never grades a placeholder as real. Distribution-valued inputs arrive as **keyed machinery evidence**; the graded summary is **derived inside the gate**, so a scalar cannot masquerade as computed. Built for kill-gate criteria 1/4/6a/6d/7; **new criteria inherit it by default.**
5. **State claims in bounded form — never rounded up.** Record exactly what is closed and what residue remains, with the residue's scope and remediation path (template: the DSR/PBO/P&L scalar-residue note). A commit message or status line broader than the code is itself an instance of the defect this section exists to prevent.
6. **"Harmless iff X" caveats verify X before X is relied on.** If a bound is checkable from data or config *now*, check it before the phase that depends on it opens — do not defer to "confirmed at first real run" (the square-off ≤ 15:20 assumption, which was false and checkable from the data manifest).
7. **Every "done" claim carries call-site evidence** — `file:line` of the **call**, not the definition — here and in `PROGRESS.md`. Reviews reject "done" without it.

## Project-Specific Inviolable Rules
These override convenience and "just make it work."

1. **The kill-gate is sacred.** No strategy is called a real edge without passing **all seven** kill-gate criteria (Part III). No tweaking-until-it-passes — that is overfitting, and every variant it spawns is charged to the honest trial ledger the Deflated Sharpe draws on. **Most strategies should die at the gate. That is success, not failure** — an honest negative is the correct, valuable outcome of a study.
2. **Point-in-time correctness, always.** No feature or signal may use data unavailable at decision time. A decision on bar *t*'s close executes at bar *t+1*'s open — identically in backtest logic everywhere. All normalization uses trailing/expanding windows only. Leakage tests run in CI.
3. **Costs are always modeled.** No gross-only backtests. The full Indian intraday cost model (brokerage, STT, exchange charges, GST, stamp duty) plus realistic slippage applies to every simulation.
4. **Honest, cumulative *effective* trial counting.** Every strategy variant ever run — including discarded ones — is charged to the ledger, and the Deflated Sharpe is deflated by the **effective number of *independent* trials**, obtained by **clustering correlated variants — never the raw backtest count**. **The ledger stores each trial's realized return stream (equivalently, a correlation structure over trials), not merely an integer counter.** Highly correlated variants — a one-at-a-time parameter sweep (MA-20/21/22/23), or any set of near-duplicates — are **not** independent trials and must not be counted as such: clustering collapses such a group to a small effective count, while genuinely distinct strategies each contribute roughly their own weight. Feeding a raw integer *N* to the DSR would over-deflate the entire program and **kill strategies that held a real edge** — that failure mode is forbidden. **What the ledger records:** one appended row per evaluation — each distinct parameter set, each meta-model variant, each re-run after any spec change — carrying that trial's return stream, never edited or deleted. **The ledger is program-wide and persists across every session and phase** (it is *not* reset per session or per phase); a study in Phase 4 is deflated by the effective count over all trials since Phase 3.1. It lives in durable storage (`research/trials/`), is machine-maintained, and the effective-*N* it yields is never a hand-typed literal. Testing ~20 strategies is heavy multiple-testing; this cluster-adjusted correction is the whole point of doing it honestly.
5. **Kite historical candle data is the only data source.** No paid alternative feeds, no scraped fundamentals, no live depth in the research program (see Part II data policy). If a strategy *requires* data Kite historical does not provide, it is out of scope and is recorded as data-gated, not faked.
6. **Honesty about outcomes.** "High stable profit" is not a goal and is not promised. Over 90% of retail F&O traders lose money — that is the operating reality. The achievable target is to *discover whether any of these strategies holds a small, real, cost-surviving edge*, and to say so plainly either way. The system must be able to conclude "none of them do."
7. **Build in dependency order; respect the gates.** The research apparatus (Phases 0–2) is built before any strategy is tested (Phases 3–4). Do not proceed past a failed gate.
8. **Ground every decision in the reference docs, and cite them.** **Part III is self-contained and sufficient to build every phase from.** The `docs/deep_dives/` files are *optional expanded reference* — consult them when a phase needs more depth, but their absence or stub state never blocks a phase (Part III governs). Before building, read the relevant Part III section; cite it in notes and PRs. Only if Part III itself is genuinely silent or self-contradictory on a decision do you STOP and surface it with options — do not guess.

# PART II — SYSTEM OVERVIEW & LOCKED DECISIONS

## What we are building
A research apparatus that ingests **Kite historical candle data** for liquid NSE cash equities, computes point-in-time features and classic technical indicators, expresses a fixed slate of **rule-based intraday strategies** as `StrategySpec`s, and runs each one through a rigorous anti-overfitting validation harness (purged CV, CPCV, Deflated Sharpe, PBO, honest costs, next-bar-open fills, robustness battery, seven-point kill-gate). The deliverable is an honest **trade / don't-trade verdict per strategy** plus a written research paper.

**This is a research-only program.** No live trading, no order placement, no capital at risk. Execution, OMS, and live infrastructure are explicitly **out of scope** and deferred (Part IV, "Beyond research") — gated on a strategy actually clearing the kill-gate. We do not build a live system around an edge that may not exist.

**Build the machine, then run the experiments.** The first thing built is the research apparatus itself — data, features, and the validation harness (Phases 0–2). Only once that honest machine exists and its gate passes do the strategies get tested through it (Phases 3–4). This ordering is a scientific requirement, not a preference: a verdict is only as trustworthy as the harness that produced it.

## Realistic-expectations frame (keep visible)
- Base rate: SEBI studies find **>90% of retail F&O traders lose money** (≈91% in FY24; ~93% over FY22–FY24). *This program trades **cash-equity intraday**, a different population from F&O — the figure is cited as a humility anchor on retail active trading generally, not a claim about this exact strategy class. Intraday cash loss rates are also high but are not the same statistic; do not conflate the two.*
- Lookahead/leakage bias inflates backtest Sharpe substantially — several studies show correcting it roughly halving or more the apparent Sharpe of ML strategies. *(If a specific figure is quoted in the paper, cite the exact source; do not carry an unsourced number in an honesty-first document.)*
- Realistic sustainable net Sharpe is **1.0–2.0**; a backtest Sharpe **>3 is a red flag**, not a trophy.
- Testing ~20 strategies means multiple-testing is severe: a few will look good by luck. The Deflated Sharpe against an honest cumulative **effective** trial count — correlated variants clustered, so a parameter sweep is not mistaken for many independent bets — is the defense.

## Data policy — Kite historical only
- **Source:** Zerodha Kite Connect **historical candle API** — OHLCV (+OI where applicable) at 1/3/5/15/60-minute and daily intervals, up to ~10 years, under the paid Connect plan.
- **Envelope:** intraday OHLCV + volume is the entire feature substrate. **No live 5-level market depth** (that is a live-stream product, not historical) → order-book / microstructure features (OFI, depth imbalance) are **out of scope** for this program.
- **All classic strategies in the slate are OHLCV-derivable** (VWAP, pivots, Donchian, moving averages, ATR, ADX, RSI, MACD, Bollinger, gaps, opening range, volume surge) — the data constraint fits the slate.
- All access behind a `BrokerAdapter` protocol; nothing else imports the Kite SDK. The paid Connect plan is required (the free Personal API has no market data).

# PART III — TECHNICAL REFERENCE

*Enough to build from. Full detail in `docs/deep_dives/01..02`.*

## Layer 1 — Data & Features

**Goal:** a versioned, point-in-time, clean OHLCV dataset and a leakage-proof feature/indicator library, behind swappable interfaces.

**Three silent killers to prevent structurally:** lookahead leakage, survivorship bias, train/serve skew.

**Connectivity (Kite historical):** REST historical-candle endpoint (~3 req/s data limit; paginate long ranges); daily session token (manual TOTP seed once/day per SEBI); paid Connect plan required. All behind a `BrokerAdapter` protocol — nothing else imports the SDK.

**Storage:** Parquet on local disk, partitioned by symbol/date, immutable raw + a derived/adjusted layer, all behind a `Repository` interface. Corrections become new versions, never silent mutations.

**Hygiene jobs (each idempotent, tested, logged):** NSE calendar + IST timestamps + pre-open/close tagging; corporate-action adjustment (store raw *and* adjusted); survivorship control via point-in-time index constituents (include delisted/renamed); bad-tick filtering (log every correction, never silently mutate); gap detection; liquidity screen (high ADV, tight spread); exclude ESM/T2T names.

**Feature & indicator families (all point-in-time pure functions, OHLCV-derived):**
- price/return transforms; realized volatility, ATR, range estimators (Parkinson/Garman-Klass);
- **VWAP** (intraday, cumulative) and VWAP-deviation; **pivot points** (classic/Fibonacci/Camarilla); **Donchian channels**; **moving averages** (SMA/EMA) and **adaptive MAs** (KAMA/AMA); **ADX**/DMI; **RSI**; **MACD**; **Bollinger Bands** & %B; **ATR-based bands/stops**;
- **opening range** (configurable window), **gap** measures, **volume surge / relative volume**, momentum & pullback measures, candlestick/pattern primitives (for bull-flag);
- time-of-day cyclical encodings; cross-sectional ranks (sector-neutral); regime features.

Prefer **TA-Lib** for standard indicators (avoid hand-rolled indicator bugs); hand-roll only what TA-Lib lacks, with tests.

**Normalization:** trailing/rolling or cross-sectional only; robust scaling; winsorize.

**Feature-store contract:** `compute_features(symbol, asof) → versioned vector`, with a CI test asserting vectorized == incremental output (the skew tripwire) and leakage tests.

## Layer 2 — Research & Validation

**Goal:** honest verdicts. *Build the validation harness before any strategy.*

**Strategy specification:** a `StrategySpec` Protocol describing a rule-based strategy as **event → entry → exit/holding → position/weight**, producing the per-period position/return series the validation engine consumes. Each of the 20 strategies (Part V) is one thin `StrategySpec`; the validation engine is written **once** and reused unchanged for all of them.

**Meta-labeling (the only ML in this program — optional, bounded, gated):** ML is *not* the signal generator here. Its one defensible role is **meta-labeling** (López de Prado): the rule-based `StrategySpec` decides the **side**; a small model (LightGBM/logistic, calibrated) decides **bet / no-bet and size** given context (ATR, time-of-day, volume, regime), tuned for precision — the natural home for the ~0.15% cost hurdle. It runs (Phase 4.5) **only** on a strategy whose rule-based version already shows a *gross* edge worth filtering; a meta-model that turns a losing rule into a winner is overfitting, not alpha. Every meta-model variant is charged to the cumulative trial ledger. The primary/secondary split is López de Prado's; see `docs/deep_dives/02`.

**Backtester realism (non-negotiable):**
- **Next-bar-open fills:** decide on bar *t*'s close, fill at bar *t+1*'s open. Never same-bar.
- **Full Indian cost model:** brokerage (lower of ~0.03% / ₹20 per order), STT (~0.025% sell-side intraday), exchange txn (~0.003%/side NSE), SEBI turnover fee (₹10/crore ≈ 0.0001%), GST (18% on brokerage + exchange txn + SEBI), stamp (~0.003% buy-side) → ~0.12–0.20% round trip intraday. **No DP charge** — that applies to delivery (CNC) sells only, not MIS intraday. All rates live in `config/costs.yaml` (statutory rates drift; never hard-code them in the backtester).
- **Slippage:** 0.05–0.20%, size/liquidity-aware; widen in stress.
- **Intraday square-off** at the configured session end (no overnight carry for MIS strategies).

**Validation (two questions, two tools):**
- *Is the edge real?* → **purged k-fold + embargo** (**embargo ≥ the max label/holding horizon**; because every position squares off intraday the horizon is ≤ 1 session, so a **1-trading-day embargo** fully removes overlap leakage — pin it, don't leave it implicit); **CPCV** (combinatorial purged CV: N groups, k test → C(N,k) splits → φ = C(N,k)·k/N paths; judge the *distribution* of path-Sharpes — narrow & positive = robust, wild variance = fragile); **Deflated Sharpe Ratio** (corrects for the **effective number of independent trials**, skew, kurtosis, length — fed by the honest trial ledger of per-trial return streams, correlated variants clustered); **PBO via CSCV**; the DSR significance bar (≥ 0.95) is kill-gate criterion 2, subsuming the classic t-stat ~3.0 hurdle.
- *What would live feel like?* → **walk-forward** with the full cost model + slippage + next-bar-open fills.

**Robustness battery:** parameter sensitivity, Monte Carlo trade shuffle, noise injection, cross-symbol validation, two-engine reconciliation (a vectorized screen vs the event-driven backtester agreeing to tolerance).

**Honest trial ledger (effective, not raw):** a machine-maintained, program-wide store of **every strategy variant's realized return stream** (including discarded ones), persisted across all sessions and phases. The DSR is deflated **not by the raw variant count but by the effective number of independent trials**: cluster the trials by P&L correlation, and let each cluster contribute an effective count reflecting its internal correlation — a tight cluster of near-duplicate variants (a one-at-a-time parameter sweep) contributes far less than its member count, while genuinely distinct strategies each contribute ~their own weight. This is the same cluster-adjusted effective-trial-count pattern as López de Prado's covariance/clustering treatment. The effective-N feeds the DSR automatically; no caller may pass a literal N.

**THE KILL-GATE (all seven; every threshold is a single pre-committed number, fixed in `config/` before the first run — never a range, never adjusted to rescue a result).** A threshold expressed as a range is an invitation to pick the lenient end when the strict end fails; that is the exact overfitting Inviolable Rule 1 forbids, so each criterion below pins one value:
1. **CPCV median path-Sharpe > 1.0**, net of costs, on the **fixed Sharpe convention** below.
2. **DSR ≥ 0.95** — the Deflated Sharpe probability that the true Sharpe exceeds the multiple-testing-adjusted benchmark, computed against the live **effective** trial count (correlated variants clustered; never a raw count, never a hand-typed N).
3. **PBO < 0.20** via CSCV. (Single pinned value; 0.20, not "0.2–0.5.")
4. **Narrow, positive CPCV distribution:** ≥ 90% of reconstructed paths have a positive Sharpe **and** the 10th-percentile path-Sharpe ≥ 0 (no deeply negative path). Both must hold.
5. **P&L not concentrated:** profit factor ≥ 1.3 **and** the top-5 winning trades contribute < 40% of gross profit **and** per-trade expectancy exceeds the modeled round-trip cost.
6. **Survives the robustness battery**, each sub-test with a pinned pass bar: parameter sensitivity (net Sharpe stays > 0.5 across ±1 config step of every parameter — no cliff); Monte-Carlo trade-shuffle (real net Sharpe beats ≥ 95% of shuffles); noise injection (edge survives realistic OHLC perturbation); cross-symbol (net-positive on a majority of held-out symbols); two-engine reconciliation (vectorized vs event-driven agree within tolerance).
   - **Amendment (P2.5 implementation note):** the "Monte-Carlo trade-shuffle" is implemented as a **sign-flip** null (`monte_carlo_sign_flip`) — each trade's return is multiplied by a random ±1 — because the Sharpe ratio (mean/σ) is *invariant to trade ordering*, so a literal order-shuffle is a no-op that tests nothing. The sign-flip is the meaningful "no directional edge" null the criterion intends. Recorded here as a deliberate amendment, not a silent substitution.
7. **Edge stable across pre-defined regimes:** partition the sample by year **and** by volatility/trend regime (fixed buckets, defined before running); require median path-Sharpe > 0 in **every** bucket and > 0.5 in a majority, and confirm that dropping the single best bucket leaves the strategy net-positive (not one bull run).

Fail any one → the strategy dies, and that honest negative is recorded in the paper. Thresholds live in `config/killgate.yaml`; changing any of them invalidates every prior verdict and re-runs the slate.

**Sharpe convention (pin before Phase 2 — a bare "Sharpe" is meaningless intraday):** every Sharpe is **annualized by √(its REALIZED frequency)** — the strategy's actual number of return observations per operating year (`realized_periods_per_year` = trades ÷ operating-span-years), computed per study from the backtest. Scaling is thus on **in-market (position-held) events, not calendar time**, and the identical convention is applied to every study so P3.1 and P3.13 are comparable on their true trade rates.
- **Amendment (P3.1 correction):** this REPLACES the original **fixed** `sharpe.periods_per_year` calendar constant (18750 = all 5-min bars in a year). That constant **contradicted the "in-market, not calendar" rule stated in this very clause** — it annualized a few-hundred-trades-a-year strategy as though it traded ~18750 times a year, inflating the Sharpe-magnitude kill-gate criteria (1/4/6a/7) toward false passes across the catalog. The drift was found by source audit and corrected **before the first catalog verdict**; `sharpe.periods_per_year`/`sharpe.basis` are removed from config (the annualization is now derived, not pinned). Because the factor is now data-dependent, a base too thin to annualize stably (`< evidence.min_base_observations` trades) returns **INSUFFICIENT** rather than a lucky small-sample Sharpe. Recorded as a deliberate amendment, not a silent substitution.

## PHASE 4.5 — Optional Meta-Labeling *(gated; skip entirely if nothing qualifies)*
*(one full session — runs only on strategies that show a gross edge in Phase 3/4)*

This is the **only** ML in the program, and it is conditional. Its purpose is to sharpen a genuine gross edge by skipping bad instances — never to resurrect a losing rule.

**Entry condition (hard gate):** a strategy from Phase 3/4 shows a **gross (pre-filter) edge worth filtering** — e.g., positive gross expectancy and a CPCV median path-Sharpe that is meaningfully positive *before* the meta-model, even if the net (post-cost) result fell short. A rule with no gross edge does **not** qualify; do not apply a meta-model to it.

- **P4.5.1 — Meta-label dataset.** For each qualifying rule signal, build the primary-side labels + triple-barrier outcomes (bet succeeded / failed), with concurrency/uniqueness sample weighting (non-IID labels). **Done when:** labels reconstruct correctly on synthetic paths; weighting tested.
- **P4.5.2 — Meta-model (bet/no-bet + size).** A small calibrated classifier (LightGBM or logistic) predicting P(bet succeeds); tuned **only under purged CV**; probabilities calibrated (isotonic/Platt); size from the calibrated probability (fractional-Kelly-capped). Feature set is context only (ATR, time-of-day, volume, regime) — never lookahead. **Done when:** trained under purged CV; calibrated; importances via permutation/SHAP (not MDI); every variant logged to the trial ledger; tested.
- **P4.5.3 — Re-judge through the kill-gate.** Run the meta-labeled strategy through the **same** seven-point kill-gate on cost-inclusive data; record the verdict and the delta vs the un-filtered rule. **Done when:** the kill-gate report exists; the paper's meta-labeling section is written; if the meta-model failed to improve net path-Sharpe, it is discarded and that is recorded.

**Guardrails:** one qualifying strategy at a time; the meta-model may only *reduce* trading (skip instances), never invent new entries; if the net improvement isn't there, the honest result is "meta-labeling did not help," recorded as such.

**GATE 4.5:** every qualifying strategy has a meta-labeled verdict recorded (improved / no-improvement); the trial ledger reflects all meta-model variants (each charged toward the effective count). Tag `gate-4.5-meta`. *(If no strategy qualified, this phase is skipped and noted as such — that is a valid outcome.)*

- **Reference:** Part III Layer 2 (meta-labeling); `docs/deep_dives/02`.
